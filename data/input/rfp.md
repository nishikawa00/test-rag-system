## 2. 背景・目的

現行の与信照会システムは2010年に稼働開始し、以下の課題を抱えている。

### 2.1 現行課題

1. **審査リードタイムの長さ**: 信用情報照会が夜間バッチのため、結果反映が翌営業日となり、最短でも1〜2営業日かかる
2. **通知の遅延**: 審査結果通知が1日2回バッチのため、顧客への連絡が遅れる
3. **手動プロセスの残存**: 紙書類確認や手動データ入力が残っており、人的ミスのリスクがある
4. **システム老朽化**: オンプレ・レガシー技術（JSP）による保守コスト増大
5. **スケーラビリティ不足**: 加盟店数増加に伴い、ピーク時の応答速度が劣化している

## 3. 要求事項

### 3.1 アーキテクチャ要件

- クラウドネイティブ（AWS または Azure）での構築
- マイクロサービスアーキテクチャの採用
- REST API による外部システムとの連携基盤の整備
- CI/CDパイプラインの導入

### 3.2 機能要件

#### 3.2.1 リアルタイム与信照会

- 加盟店からのAPIリクエストに対し、**リアルタイム**で与信照会結果を返すこと
- 目標応答時間: **1秒以内**（P99）

#### 3.2.2 信用情報機関のリアルタイム連携

- 現行のバッチ連携から**リアルタイムAPI連携**へ移行
- 複数の信用情報機関（TDB, CIC等）との連携を想定した設計
- キャッシュ戦略の実装（照会結果の有効期間設定）

#### 3.2.3 自動審査エンジン

- ルールベースの自動審査機能を実装
- 自動承認・自動否認・人的審査（エスカレーション）の3段階フロー
- 審査ルールのパラメータ設定をノーコードで変更可能にすること

#### 3.2.4 申請〜通知のリアルタイム化

- 申請受付後、即時に受付通知を送信
- 審査完了後、即時に結果を通知（メール・Webhook）
- 加盟店向けポータルに申請状況をリアルタイム表示

#### 3.2.5 与信管理機能の強化

- 与信限度額の利用状況をリアルタイムに管理
- 限度額変更の申請・承認ワークフロー（既存は手動）
- アラート機能: 利用率が閾値（例: 80%）を超えた場合に通知

#### 3.2.6 管理機能

- 審査ルールの設定・変更（管理者）
- 加盟店・与信情報の一覧管理・検索
- 審査統計レポートのダッシュボード化

### 3.3 非機能要件

#### 3.3.1 可用性

- **24時間365日稼働**（現行は平日業務時間帯のみ）
- 目標稼働率: **99.9%以上**
- RTO（目標復旧時間）: **1時間以内**
- RPO（目標復旧時点）: **15分以内**

#### 3.3.2 性能

- 与信照会API応答時間: **1秒以内**（P99）
- 同時リクエスト数: **500リクエスト/秒**に対応
- オートスケーリングによる負荷対応

#### 3.3.3 セキュリティ

- OAuth 2.0 / OpenID Connect による認証認可
- API キー管理・ローテーション
- 通信暗号化: TLS 1.3
- 個人情報・機密データの暗号化（保存データ含む）
- WAF（Web Application Firewall）の導入
- ISMS認証取得を目指した設計

#### 3.3.4 監査・ログ

- 全操作・API呼び出しの監査ログ記録
- ログ保存期間: 5年間（現行は3年）
- ログの改ざん防止機能

### 3.4 移行要件

- 現行データの完全移行（10年分の申請・与信データ）
- 移行期間中の並行稼働期間を設けること（最低2ヶ月）
- ゼロダウンタイムでの本番切替
